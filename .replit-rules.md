# Regras obrigatórias — Figma → Código (Pixel Perfect 1:1)

> Este arquivo é a **fonte da verdade** para qualquer trabalho de implementação a partir de design no Figma neste projeto.
> O agente deve **ler estas regras antes de codar** e segui-las **sem exceção**.

---

## 0. Princípio absoluto

- **NÃO** interpretar o layout.
- **NÃO** criar aproximação visual.
- **REPRODUZIR EXATAMENTE** o design do Figma.
- **Objetivo:** pixel perfect 1:1 com o PNG de referência e o frame do Figma.

O agente deve **medir, implementar, comparar e corrigir** — nunca "achar parecido".

---

## 1. Fonte da verdade

- **MCP = source of truth** (medidas, tipografia, cores, espaçamento, auto-layout, variáveis, constraints, ícones — tudo é extraído do MCP).
- **PNG = visual validation only** (usado apenas para conferência visual no fim).
- **Nunca confiar apenas em screenshots.** Sempre extrair via MCP.

Sempre extrair do Figma MCP:
- typography (font-family, size, weight, line-height, letter-spacing)
- spacing (padding, margin, gap)
- dimensions (width, height, min/max)
- auto-layout (direção, alinhamento, gap, padding)
- variables / tokens
- colors (HEX/RGBA com alpha)
- icon sizes e posições
- constraints e effects (shadow, blur, etc.)

## 2. Skills obrigatórias antes de codar

**Antes de implementar qualquer seção do Figma**, carregar e seguir, nesta ordem:

1. `.local/skills/figma-skills/figma-use/SKILL.md`
2. `.local/skills/figma-skills/figma-implement-design/SKILL.md`
3. `.local/skills/figma-skills/figma-use/references/text-style-patterns.md`
4. `.local/skills/figma-skills/figma-use/references/variable-patterns.md`
5. `.local/skills/figma-skills/figma-use/references/working-with-design-systems/wwds.md` *(quando o arquivo Figma usa tokens, componentes ou estilos compartilhados)*

Skills auxiliares (usar conforme o caso):

| Skill | Quando usar |
|---|---|
| `figma-code-connect` | Mapear componentes Figma ↔ código existente |
| `figma-create-design-system-rules` | Quando envolve criar regras de design system |
| `figma-generate-design` / `figma-generate-library` / `figma-generate-diagram` / `figma-create-new-file` / `figma-use-figjam` | Casos específicos de escrita no Figma |

**Nunca implementar uma seção do Figma sem carregar essas skills primeiro.**

Se o Figma MCP estiver com rate-limit, usar `FIGMA_TOKEN` (secret) via REST: `curl -H "X-Figma-Token: $FIGMA_TOKEN" "https://api.figma.com/v1/files/<key>/nodes?ids=<id>"`.

**Não estimar medidas que possam ser lidas via MCP/REST.**

## 3. Fluxo obrigatório antes de codar

1. Carregar as skills da seção 2 acima (figma-use, figma-implement-design, text-style-patterns, variable-patterns, +wwds se aplicável).
2. Ler o frame/seção no Figma via MCP/REST (fonte da verdade).
3. Usar o PNG **apenas como referência visual** para validação posterior.
4. Identificar a seção exata a recriar.
5. Listar os arquivos que pretende alterar.
6. Extrair e documentar as medidas principais no formato:

```txt
Nome da seção:
Frame do Figma:
Largura:
Altura:
Background:
Container:
Padding top:
Padding bottom:
Padding lateral:
Gap principal:
Tipografia:
Cards/blocos:
Ícones:
Imagens:
Responsividade:
```

---

## 4. Extração obrigatória (antes de implementar)

Para cada elemento, extrair e listar:

- `font-family`
- `font-size`
- `font-weight` (numérico — nunca `bold`)
- `line-height` (exato — nunca `normal`)
- `letter-spacing`
- Cores HEX/RGBA
- `width`, `height`
- `padding`, `margin`, `gap`
- `border-radius`
- `box-shadow` / effects
- Stroke width
- Tamanho dos ícones
- Posição dos ícones
- Alinhamento dos elementos

---

## 5. Restrições absolutas

- ❌ Não alterar outras seções, header, footer, menu ou estilos globais.
- ❌ Não refatorar o projeto.
- ❌ Não remover código existente fora do escopo.
- ❌ Não substituir componentes fora da seção atual.
- ❌ Não trocar ícones/SVGs fornecidos por bibliotecas externas (Lucide, Font Awesome, Heroicons, etc.).
- ❌ Não usar valores aproximados quando o Figma fornece o valor exato.
- ❌ Não usar `font-weight: bold` (usar valor numérico).
- ❌ Não usar `line-height: normal`.
- ❌ Não usar `padding`, `margin` ou `gap` genéricos.
- ❌ Não usar `auto`, `inherit` ou padrões do navegador quando o Figma tem valor explícito.
- ❌ Não usar `space-between` se o Figma usa `gap` fixo.
- ❌ Não simplificar o layout.

---

## 6. Tipografia

```css
font-family: <valor-do-figma>;
font-size: <valor-em-px>;
font-weight: <valor-numérico>;
line-height: <valor-exato>;
letter-spacing: <valor-exato>;
```

Se a fonte não existir no projeto, **verificar primeiro se já está importada**. Caso não esteja, importar **somente o necessário** sem impactar outras seções.

Convenções deste projeto (Provider Mais Fibra):
- **Nunito** (600/700) → header/menu
- **Montserrat** → body/restante do site
- **Amino Black** → números de velocidade dos planos
- **Nexa Black** → preços

---

## 7. Ícones e assets

- Usar **somente** os ícones/SVGs/PNGs fornecidos pelo usuário (geralmente em RAR com nome `nome+sessao+tamanhoPx`).
- Nunca substituir por biblioteca externa.
- Conferir: tamanho, cor, stroke width, alinhamento, distância entre ícone e texto.
- Cada ícone deve ser validado contra o PNG.

---

## 8. Layout, espaçamento e Auto Layout

Traduzir Auto Layout do Figma para CSS de forma fiel:

```css
display: flex;
flex-direction: <row|column>;
align-items: <valor-do-figma>;
justify-content: <valor-do-figma>;
gap: <valor-do-figma>;
padding: <valor-do-figma>;
```

Usar `grid` apenas quando a estrutura visual realmente exigir.

---

## 9. Responsividade

1. Implementar **desktop fiel ao PNG/Figma primeiro**.
2. Adaptar para tablet e mobile **sem quebrar o desktop**.
3. Preservar: hierarquia visual, espaçamentos proporcionais, leitura, alinhamento de ícones, ordem dos elementos, tamanhos.

---

## 10. Validação obrigatória depois de implementar

Após codar, comparar visualmente o resultado com o PNG (visual validation) e verificar:

- [ ] fonte correta
- [ ] font-size correto
- [ ] font-weight correto
- [ ] line-height correto
- [ ] cor correta
- [ ] padding correto
- [ ] gap correto
- [ ] posição dos elementos
- [ ] tamanho dos ícones
- [ ] ícones corretos
- [ ] altura da seção
- [ ] largura do container
- [ ] alinhamento dos cards
- [ ] border-radius
- [ ] sombras
- [ ] responsividade

Encontrou diferença? **Corrigir antes de responder.**

---

## 11. Processo de correção

- Não refazer tudo.
- Não mexer em outras seções.
- Corrigir somente a diferença detectada.
- Comparar de novo.
- Repetir até ficar visualmente equivalente.

---

---

## 13. Regra final

**Não finalizar enquanto a seção estiver apenas "parecida".**
Finalizar somente quando estiver **visualmente equivalente** ao PNG/Figma e **sem alterações fora do escopo**.
