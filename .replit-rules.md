# Regras obrigatórias — Figma → Código (Pixel Perfect 1:1)

> Este arquivo é a **fonte da verdade** para qualquer trabalho de implementação a partir de design no Figma neste projeto.
> O agente deve **ler estas regras antes de codar** e segui-las **sem exceção**.

---

## 0. Princípio absoluto

- **NÃO** interpretar o layout.
- **NÃO** criar aproximação visual.
- **REPRODUZIR EXATAMENTE** o design do Figma.
- **Objetivo:** pixel perfect 1:1 com o PNG de referência e o frame do Figma.

O agente deve **medir, implementar, comparar e corrigir** — nunca "achar parecido".

---

## 1. Uso obrigatório das Skills do Figma MCP

Sempre que receber um link/nó do Figma, **antes de codar**, abrir e seguir a skill apropriada em `.local/skills/figma-skills/`:

| Skill | Quando usar |
|---|---|
| `figma-implement-design` | Traduzir nó/frame do Figma em código (skill principal) |
| `figma-use` | Ler o arquivo Figma via API/MCP |
| `figma-code-connect` | Mapear componentes Figma ↔ código existente |
| `figma-create-design-system-rules` | Quando envolve tokens/design system |
| `figma-generate-design` / `figma-generate-library` / `figma-generate-diagram` / `figma-create-new-file` / `figma-use-figjam` | Casos específicos |

Se o Figma MCP estiver com rate-limit, usar `FIGMA_TOKEN` (já salvo como secret) com a REST API: `curl -H "X-Figma-Token: $FIGMA_TOKEN" "https://api.figma.com/v1/files/<key>/nodes?ids=<id>"`.

**Não estimar medidas que possam ser lidas via MCP/REST.**

---

## 2. Fluxo obrigatório antes de codar

1. Ler o frame/seção no Figma via MCP/REST.
2. Analisar o PNG de referência fornecido.
3. Identificar a seção exata a recriar.
4. Listar os arquivos que pretende alterar.
5. Extrair e documentar as medidas principais no formato:

```txt
Nome da seção:
Frame do Figma:
Largura:
Altura:
Background:
Container:
Padding top:
Padding bottom:
Padding lateral:
Gap principal:
Tipografia:
Cards/blocos:
Ícones:
Imagens:
Responsividade:
```

---

## 3. Extração obrigatória (antes de implementar)

Para cada elemento, extrair e listar:

- `font-family`
- `font-size`
- `font-weight` (numérico — nunca `bold`)
- `line-height` (exato — nunca `normal`)
- `letter-spacing`
- Cores HEX/RGBA
- `width`, `height`
- `padding`, `margin`, `gap`
- `border-radius`
- `box-shadow` / effects
- Stroke width
- Tamanho dos ícones
- Posição dos ícones
- Alinhamento dos elementos

---

## 4. Restrições absolutas

- ❌ Não alterar outras seções, header, footer, menu ou estilos globais.
- ❌ Não refatorar o projeto.
- ❌ Não remover código existente fora do escopo.
- ❌ Não substituir componentes fora da seção atual.
- ❌ Não trocar ícones/SVGs fornecidos por bibliotecas externas (Lucide, Font Awesome, Heroicons, etc.).
- ❌ Não usar valores aproximados quando o Figma fornece o valor exato.
- ❌ Não usar `font-weight: bold` (usar valor numérico).
- ❌ Não usar `line-height: normal`.
- ❌ Não usar `padding`, `margin` ou `gap` genéricos.
- ❌ Não usar `auto`, `inherit` ou padrões do navegador quando o Figma tem valor explícito.
- ❌ Não usar `space-between` se o Figma usa `gap` fixo.
- ❌ Não simplificar o layout.

---

## 5. Tipografia

```css
font-family: <valor-do-figma>;
font-size: <valor-em-px>;
font-weight: <valor-numérico>;
line-height: <valor-exato>;
letter-spacing: <valor-exato>;
```

Se a fonte não existir no projeto, **verificar primeiro se já está importada**. Caso não esteja, importar **somente o necessário** sem impactar outras seções.

Convenções deste projeto (Provider Mais Fibra):
- **Nunito** (600/700) → header/menu
- **Montserrat** → body/restante do site
- **Amino Black** → números de velocidade dos planos
- **Nexa Black** → preços

---

## 6. Ícones e assets

- Usar **somente** os ícones/SVGs/PNGs fornecidos pelo usuário (geralmente em RAR com nome `nome+sessao+tamanhoPx`).
- Nunca substituir por biblioteca externa.
- Conferir: tamanho, cor, stroke width, alinhamento, distância entre ícone e texto.
- Cada ícone deve ser validado contra o PNG.

---

## 7. Layout, espaçamento e Auto Layout

Traduzir Auto Layout do Figma para CSS de forma fiel:

```css
display: flex;
flex-direction: <row|column>;
align-items: <valor-do-figma>;
justify-content: <valor-do-figma>;
gap: <valor-do-figma>;
padding: <valor-do-figma>;
```

Usar `grid` apenas quando a estrutura visual realmente exigir.

---

## 8. Responsividade

1. Implementar **desktop fiel ao PNG/Figma primeiro**.
2. Adaptar para tablet e mobile **sem quebrar o desktop**.
3. Preservar: hierarquia visual, espaçamentos proporcionais, leitura, alinhamento de ícones, ordem dos elementos, tamanhos.

---

## 9. Validação obrigatória depois de implementar

Após codar, comparar visualmente o resultado com o PNG (usando screenshot do app) e verificar:

- [ ] fonte correta
- [ ] font-size correto
- [ ] font-weight correto
- [ ] line-height correto
- [ ] cor correta
- [ ] padding correto
- [ ] gap correto
- [ ] posição dos elementos
- [ ] tamanho dos ícones
- [ ] ícones corretos
- [ ] altura da seção
- [ ] largura do container
- [ ] alinhamento dos cards
- [ ] border-radius
- [ ] sombras
- [ ] responsividade

Encontrou diferença? **Corrigir antes de responder.**

---

## 10. Processo de correção

- Não refazer tudo.
- Não mexer em outras seções.
- Corrigir somente a diferença detectada.
- Comparar de novo.
- Repetir até ficar visualmente equivalente.

---

## 11. Resposta final obrigatória

Sempre encerrar a entrega com:

```txt
Arquivos alterados:
- ...

O que foi implementado:
- ...

Medidas usadas do Figma:
- ...

Ajustes de responsividade:
- ...

Validação visual:
- Fonte conferida
- Espaçamentos conferidos
- Ícones conferidos
- Alinhamentos conferidos
- Desktop conferido
- Mobile conferido

Observações:
- ...
```

Se algum dado do Figma não puder ser lido pelo MCP, informar claramente e usar o **PNG como referência principal**.

---

## 12. Regra final

**Não finalizar enquanto a seção estiver apenas "parecida".**
Finalizar somente quando estiver **visualmente equivalente** ao PNG/Figma e **sem alterações fora do escopo**.
